#2 — Reports page split from Dashboard

Status: in discussion · Locked: Boundary · Created: 2026-05-04 · Scope: No schema, 4 new endpoints

Dashboard = today/now/urgency. Reports = cross-module analysis & history. "Everything About a Family Member" lives in Reports, not Kids. This draft assigns each existing dashboard widget to one side or the other.

The boundary

Dashboard — today / now / urgency

  • Inventory expiring alerts
  • Todos due today / overdue
  • HSA missing receipts count
  • Career certs expiring soon
  • Calendar upcoming events
  • Daily log latest entry
  • Subscriptions renewing this week
  • Bell-icon notifications badge

Reports — analysis / completeness / history

  • Net worth snapshot & history
  • Inventory breakdown by room/category
  • Medical history & medication summary
  • Property estimated value & trend
  • HSA / FSA IRS reimbursement (added v140)
  • Document expiry timeline
  • Data quality / completeness scorecard
  • Everything About a Family Member

Reports card grid (uses existing pattern)

Reports keeps the existing 2-zone card-grid pattern. Each card = one report. Click → opens that report. NOT the 4-zone medication card.

Open questions

30-second polling for "live record fixing"
User mentioned new-tab + 30s polling so reports refresh while you fix records in other tabs. Use case still TBD.
Which widgets stay vs move (boundary above is proposed)
Net worth — does a small "Net worth: $X" snapshot stay on dashboard with a "View report →" link, or fully move?

Build sequence

  1. Audit existing index.html widgets. Classify each.
  2. Demote analysis widgets — keep small snapshot on dashboard, move detail to Reports.
  3. Add 4 new Reports card configs to REPORT_REGISTRY.
  4. Build "Everything About a Family Member" backend endpoint /api/v1/reports/family-snapshot/:id.
  5. Build report renderer using existing reports.html pattern.
  6. (Separately) decide and implement 30s-polling pattern.

Backend changes

Four new GET endpoints (read-only, no schema):

  • /api/v1/reports/family-snapshot/:id — nested family-member view
  • /api/v1/reports/data-quality — completeness scorecard
  • /api/v1/reports/inventory-breakdown — by room, by category
  • /api/v1/reports/networth-history — point-in-time net worth series

Out of scope

  • Notifications — separate draft #3
  • Cron-driven scheduled reports
  • Email-the-report
  • PDF export (XLSX exists, sufficient)
