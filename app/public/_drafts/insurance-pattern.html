#20 — Insurance / historical-tracking pattern

Status: spec ready · Priority: medium · Sources: chat 8

Generic "policy chain" pattern — one parent record, many historical versions linked by chain_id. See 2022 → 2024 → 2026 side by side. Reusable across insurance, utility rates, mortgage history, HOA fees, property tax history. Build once, instantiate per module.

Entities

  • policy_chains — id, kind, label, primary_contact_id, is_active
  • policy_versions — chain_id (FK), version_start_date, version_end_date, premium_amount, premium_frequency, deductible, coverage_limits (JSON), provider_id (FK contacts), policy_number, status, notes
  • Attachments use existing system, entity_type='policy_version'

What reuses this pattern

  • Insurance — auto, home, life, health, umbrella, renters
  • Utility rates — electric, gas, water
  • Mortgage / loan rate history
  • HOA fees year-over-year
  • Property tax history (assessed value, rate, paid)

  2. Build CRUD routes (chain has nested versions endpoint). (Claude)
  3. Build insurance module page as first consumer. (Claude)
  4. Wire "Renew" action — creates new version with end_date on prior, copies fields user can override. (Claude)
  5. Build side-by-side version comparison view. (Claude)
  6. Add Reports: active policies, expiring 60d, premium spend by year, coverage history per chain. (Claude)

  8. (Later) Wire utility rates, mortgage, HOA, property tax. (deferred)

Locked agreements

  • One source of truth: policy_chains owns the relationship, policy_versions owns the data.
  • Renewal = new version with end_date set on prior. NEVER edit a past version's facts (audit trail).
  • Provider = contacts FK, not free text.
  • Versions never delete — soft-only via policy_chains.is_active.
  • Reports filter on chain.is_active for "active" view, ignore for history.
