Slice 1 — locked design decisions (this chat)

  • Multi-condition: junction table med_medication_conditions (singular condition_id column kept, not dropped — additive only).
  • HSA ↔ entity link: polymorphic hsa_payment_links (entity_type + entity_id, allowlist enforced server-side).
  • Fill history: new med_medication_fills table.
  • New med_medications columns: brand_name, generic_name, form, drug_class, take_with_food, time_of_day, interaction_warning.
  • Hero on the card = whatever was prescribed. Subtitle shows the OTHER name ("generic version of Lipitor" if generic was prescribed).
  • Drug class column added but hidden from display when empty. Auto-lookup via RxClass API deferred to a later slice.
  • Pill imagery: user-uploaded photo via attach-lifecycle wins → falls back to Phosphor icon by form field, status-tinted. Procedural pill render dropped.
  • HSA receipt linking is bidirectional — link from med card and from HSA receipt. Plain language, no jargon.
  • family_member_id backfilled from patient text on case-insensitive match. patient column kept alive (no destructive deletes per project rules).

Risks — what could go wrong

  • Migration runner parser bug — semicolons inside SQL comments break parsing. Worked around in 118; needs proper fix in migrate.js.
  • Card system rollout — once we start applying the 4-zone card to real modules, regressions are likely. Recommend one module at a time, smoke-test each.
  • Center modals sweep — touches every page implicitly. Any module overriding .gh-drawer CSS locally will behave differently. Need audit before ship.
  • v140 display layer — currently shipping data without UI. If a power user opens API directly, they'll see empty tables. Low risk but worth noting.
  • Notifications — node-cron inside container loses in-flight ticks on restart. Acceptable trade-off but worth noting.
