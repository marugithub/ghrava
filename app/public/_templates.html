Locked design rules

  1. Title in plain English. Not "Q1 OOP HSA breakdown" — say "Medical costs by quarter."

  6. Empty state preserves the chart frame with "No data yet" placeholder (same rule as tiles).

#27 Auto-Linkers Pattern

Locked v.167 Every cross-module auto-link uses this shape: matcher → confidence → link or review-flag. Manual override always allowed.

The four-step shape

  1. Match criteria (per linker) — strict field equality + tolerance windows. No fuzzy text. Documented in auto-link-*.js at top of file.
  2. Confidence tier:
    • High — all required fields match within tolerance → auto-link silently.
    • Medium — most fields match, 1 fails → create link + set needs_review = 1.
    • Low / none — no link created.
  3. Review surface — "Needs review" pill on the originating module's list view. Click → drawer with side-by-side comparison + "Confirm link / Unlink / Adjust" buttons.
  4. Manual override always allowed — from any list view, "Link to…" picker lets Al manually create a record_links row.

Linkers locked in v.167

# Name Match criteria (HIGH = auto-link, otherwise review) Trigger
#27.1txn → hsa_paymentHIGH: account_type='HSA' OR account_name LIKE '%HSA%'. Creates hsa_payment row with you_paid = txn.amount, payment_date = txn.date, vendor = txn.merchant.On txn import-confirm + on existing-txn category change to "medical"
#27.2txn → medical_visitHIGH: txn.merchant exact-match (case-insensitive trim) contacts.name WHERE type='medical_provider' AND visit_date within ±7d of txn.date.On txn import-confirm + on visit creation
#27.3EOB → hsa_paymentHIGH: same patient (family_member_id required), amount equals your_share (±$0.50), date within ±14d. Provider/Dr match is BONUS but not required.On EOB parse + on new hsa_payment
#27.4txn → subscription (category)Subscription auto-match (v.157) already links. THIS adds: copy subscription's category to the txn IF txn.category IS NULL.Ongoing on every import + retroactive button (last 90d)

Shared infrastructure (one place, used by all linkers)

  • app/shared/auto-link.js — helper module with createLink(leftType, leftId, rightType, rightId, kind, confidence, needsReview). Writes to record_links table.
  • record_links.confidence column (NEW v.167): 'high' | 'medium' | 'low'. Default 'high' when manually created.
  • record_links.needs_review column (NEW v.167): boolean. UI surfaces as pill on entity list views.
  • GET /api/v1/links/needs-review — list endpoint for Today/Inbox surfaces.
  • POST /api/v1/links/:id/confirm — clear needs_review.
  • DELETE /api/v1/links/:id — manual unlink.

UI rule

Every list view that shows entities with auto-linker outputs (transactions, EOBs, hsa_payments) shows a "needs review" pill on rows where any linked record has needs_review = 1. Pill click opens the review drawer. No silent auto-actions without a visible undo path.

#28 Universal Attachments

Design locked v.167 · Build in v.168 One file on disk, many links. Edit once, reflect everywhere. Refcount-based deletion.

Problem

Same receipt can apply to multiple modules: a CVS purchase is an inventory item, an HSA payment, AND a transaction. Today each module stores its own attachment row pointing at its own copy of the file. Three copies of the same file means three sources of truth and inevitable drift.

Solution

One canonical attachment row + record_links rows pointing at it from each module. File on disk written once. Metadata (filename, tags, OCR text) edited once. Each link can carry a per-context label (link_kind: receipt/warranty/manual/eob) without changing the file.

Upload flow (locked)

  1. User clicks "Add attachment" from any module (inventory item, txn, hsa_payment, document, visit, etc.).
  2. File uploads → creates ONE row in attachments with the file path.
  3. System runs smart pre-check matcher against other modules using the originating context (amount, date, patient, vendor).
  4. Modal shows: file preview + originating module pre-linked + "Also relevant to" multi-select with auto-checked HIGH matches and unchecked MEDIUM matches (with "?" badge).
  5. User confirms → each checked box creates a record_links row pointing at the attachment.

Pre-check thresholds (locked)

Target module HIGH match (auto-check) MEDIUM match (unchecked + ?)
Transactionamount equals AND vendor exact AND date ±2damount equals AND date ±7d (no vendor match)
HSA paymentamount equals AND date ±2damount equals AND date ±14d
Medical visitvisit_date ±2d AND provider matches a contact in family member's care teamvisit_date ±7d
Inventory itemnone auto (need explicit user pick)item created on same date as upload

Deletion semantics (locked)

  • Module-side delete: "Remove attachment from this item" deletes only the record_links row. File stays.
  • Confirm-with-holder-list: if the attachment has 2+ links, the unlink dialog shows: "This file is also linked to: HSA payment $32 (May 13), Transaction CVS $32. Continue removing from inventory only?"
  • Refcount drops to 0: attachment moves to soft-deleted (30-day undo window, same as everything else).
  • Hard delete (Settings → Recently Deleted): removes file from disk.

Schema (v.168 migration)

  • record_links gains: attachment_id INTEGER REFERENCES attachments(id) ON DELETE SET NULL (for attachment-bearing links), link_kind TEXT (receipt/warranty/manual/eob/source_pdf/other).
  • attachments gains: refcount INTEGER (denormalized count of record_links rows pointing here; trigger-maintained for safety), soft_deleted_at DATETIME for the 30-day window.
  • Existing attachments(entity_type, entity_id) columns stay populated for backward compat. New uploads write BOTH a record_links row AND the legacy columns until the migration backfill runs.
  • Backfill script: for each existing attachment row, INSERT a record_links row with matching entity_type/entity_id, link_kind='legacy_migrated'. Idempotent.

Affected modules (14)

Inventory, Documents, Medical (visits + meds + conditions), HSA, FSA, Vehicles, Property, Career, Books, Wardrobe, Perfume, Subscriptions, Insurance, Daily Log. Every "Add attachment" UI on these modules switches to the shared upload dialog in v.168.

New endpoints

  • GET /api/v1/attachments/:id/links — list every entity linked to this file
  • POST /api/v1/attachments/:id/links — add a link (body: entity_type, entity_id, link_kind)
  • DELETE /api/v1/attachments/:id/links/:linkId — remove a link; if refcount=0 → soft-delete file
  • POST /api/v1/attachments/match-suggestions — smart pre-check matcher (body: amount, date, vendor, family_member_id, source_module) → returns suggested HIGH/MEDIUM matches

v.168 build sequence

  1. Schema migration (additive to record_links + attachments)
  2. New endpoints + matcher
  3. Shared upload dialog component (`/js/universal-attach.js`)
  4. Migrate Inventory + HSA + Medical first (highest-value path)
  5. Migrate remaining 11 modules
  6. Backfill script (manual run after deploy)
  7. Update help.html with new commands + Settings UI to view "shared attachments"

#29 Pending Items Report

Locked v.171 One report for every bank charge that hasn't been linked yet. Pick once, the rule remembers. Three buttons every row: assign / skip 30d / not a [module] expense.

Where it lives

/reports.html?tab=pending — first tab after Overview. Tab badge shows the open count. Two views (locked):

  • List view (default) — dense one-line rows, picker expands inline below the row. Best for processing through many items.
  • Grid view — same items as 2–4 column tiles, picker opens in a modal on tap. Best for a quick visual scan.

View toggle via the standard GH_VIEW.init('pendingViewToolbar', 'pending', cb, { views: ['grid','list'], defaultView: 'list' }). State persists per device in localStorage.

Row anatomy (list view)

Left to right: module icon (28×28, neutral background) · plain-English prompt (e.g. "Which car was this fuel for?") · meta line (merchant · amount · date · suggested record if any) · chevron. Whole row is the click target; tap expands the picker drawer below it.

Tile anatomy (grid view)

Card with: head row (28×28 icon, uppercase module eyebrow, date right-aligned), prompt (14px medium), meta (12px secondary, merchant + amount), full-width primary CTA at the bottom. Tap opens the same picker as the list, but in a centered modal.

Filter chips (locked)

Above the list: All · Vehicles · Medical · Inventory · Subscriptions · HSA receipts · Career. Each shows its current count (· N). Active chip is filled-dark; inactive chips are pill-bordered.

Picker drawer (locked)

  1. Picker label in plain English per module ("Assign to" / "Which prescription?" / "Link to inventory item" / etc.).
  2. Option pills — one per candidate record from the relevant module. Suggested record (when backend supplies one) appears first. Selected pill turns accent-colored.
  3. "Remember rule" checkbox — checked by default; the merchant-token is bolded inline so the user can read what they're agreeing to: "Always send SHELL charges here".
  4. Three actions, every row, every module:
    • Primary — plain-English action per module ("Pick a car" / "Enter cost" / "Track it" / "Link to item" / "Upload receipt" / "Link to cert"). Disabled until a pill is selected.
    • Skip for now — snoozes 30 days. The row disappears and won't return until the snooze window expires.
    • Not a [module] expense — permanent dismiss. Stops the row coming back ever. Confirm dialog before commit.

The asterisk pattern (locked)

Any card on any page that shows a derived number wraps it with a .gh-pending-target span containing a .gh-pending-host. On page-ready, GhAsterisk.scan() hits GET /api/v1/pending/asterisk?card=…&record_id=… for each target. Colors: amber (some items pending, number still mostly reliable), red (at least one underlying record missing a required field). Click the asterisk → jumps to the Pending Items tab pre-filtered to that module.

Empty state (locked)

✨ + "Nothing needs your attention right now." + "New bank charges will appear here after the next import." No marketing copy. No prompts to import.

API surface

  • GET /api/v1/pending · GET /api/v1/pending/counts · POST /api/v1/pending/link · POST /api/v1/pending/skip · POST /api/v1/pending/dismiss · GET /api/v1/pending/rules · DELETE /api/v1/pending/rules/:id · GET /api/v1/pending/asterisk
  • Per-tx ↔ record link goes in shared record_links table (left_type='transaction'). The SHARED-LNK lock is honored.
  • New tables: tx_link_rules (merchant→record auto-apply), pending_dismissals (skip/dismiss markers).
