#18 — .env.secrets migration

Status: in discussion · Priority: high (security) · Sources: chat 8, chat 9

Move secrets out of app_config (which sits inside the DB backup) into a file Docker mounts and Node reads at startup. Backup never sees keys again. User raised this 3× across chats 8 and 9. Design ready, never built.

What moves vs stays

Moves to .env.secretsStays in app_config
google_client_idgoogle_access_token (rotates)
google_client_secretgoogle_refresh_token (rotates)
google_books_api_keylast-sync timestamps
app_password_hashsync preferences, feature flags
any future API keyper-user prefs

Actions

  1. Decide mount path inside container — likely /app/.env.secrets via bind mount. (user)
  2. Add dotenv to package.json. (Claude)
  3. Wire load at server.js boot. (Claude)
  4. Refactor every secret-read site (oauth.js, books key handler, password check) to read process.env first, fall back to app_config. (Claude)
  5. Update Settings UI: show "configured via .env.secrets" instead of editable input when env value present. (Claude)
  6. Migration to NULL out moved fields in app_config. (Claude)
  7. Add existing keys to /share/Backups/ghrava/.env.secrets on NAS. (user)

Locked agreements

  • File path: same .env.secrets template user already has on NAS.
  • Read order: process.env wins over app_config (env overrides DB).
  • Settings UI shows status only when env present, never the value.
  • Backup of lifetracker.db must contain zero secrets after migration.
  • Rotating tokens (access / refresh) STAY in DB — they change too often.

Open questions

What happens if .env.secrets is missing?
Lean: log warning, fall back to DB. Smoother first-time setup.

Risk

Touching auth (password hash) is highest-risk. Land Google keys first, password hash second after smoke test passes.

Out of scope

  • Encrypting the secrets file at rest (separate concern)
  • Vault integration (overkill for one-household app)
