#10 — Data quality / completeness checker

Status: spec ready · Priority: medium · Sources: chat 3, chat 5

Surfaces records with missing important fields. Different from Orphan Check (which is structural FK integrity). This is field-level. Lives inside Reports module per chat 5 decision.

How records get flagged (two inputs)

Triggerreview_reason example
Migration restructures a field "System redesign: location type restructured (2026-03-16)"
CSV import field mismatch "CSV import: category 'Health & Beauty' not matched to known value"
CSV import missing required "CSV import: location field was blank"
Bulk update / rename "Bulk update: inventory category renamed"

Entities (mostly already built)

Three columns exist on most taggable tables:

  • needs_review INTEGER NOT NULL DEFAULT 0
  • review_flagged_at DATETIME
  • review_reason TEXT

flagRecord() helper exists in shared/ but per chat 7 audit is never called by any route. Wire it everywhere.

  6. Build Reports card "Data Quality" calling that endpoint. (Claude)
  7. Wire per-row "Fix" action opens record drawer in new tab. (Claude)
  8. Wire per-row "Mark reviewed" clears the flag without editing the record. (Claude)
  9. Wire bulk "Mark all reviewed" per module section. (Claude)

Locked agreements

  • Three columns: needs_review, review_flagged_at, review_reason.
  • Reason format: prefixed with source — "CSV import:", "System redesign:", "Bulk update:".
  • Old data NEVER overwritten when flagged — value preserved.
  • flagRecord() helper is the single entry point — no inline updates.
  • Lives inside Reports module (chat 5 decision), NOT Settings.
  • Per-row Fix opens record in new tab; report polls 30s for updates.

Open questions

"Important fields" definition per module
Needs per-module list. Inventory: location, category. Contacts: phone or email. Medications: dosage, patient. Defer per-module decisions.
Auto-clear flag when missing field is filled, or require manual "Mark reviewed"?
Lean: auto-clear (less friction).

Risk

flagRecord() exists but unused. Wiring it across all modules will surface hundreds of rows that should have been flagged historically but werent. Acceptable — thats reality. First-scan number will be high.

Out of scope

  • Auto-fixing (e.g. inferring missing category from name) — separate AI work
  • Cross-record validation (e.g. duplicate detection)
