Tailwind migration — internal plan

Internal working doc. Tracks every page's migration cost so I don't lose track. Do not link from the user-facing nav.

Strategy

Opportunistic. New pages built in Tailwind from day one. Existing pages stay vanilla until I'm rebuilding them anyway. Tailwind via CDN during prototyping, switch to CLI compile step when it lands in production.

Component reuse strategy

Shared components I rebuild once in Tailwind, every page benefits as it migrates:

  • Drawer — replace .drawer-overlay/.drawer with Tailwind center-modal. Reusable.
  • FAB — replace .fab/.fab-pill. Smaller scale (44px not 54px).
  • Form fields — Tailwind utility classes wrapped in .gh-input, .gh-select via plugin or @apply.
  • Buttons — primary / ghost / icon. Tailwind utility classes.
  • Tabs / segmented control — pattern from reference design.
  • Cards — Card #1 v5 (med), Card #6 (cond), card patterns rebuilt as Tailwind component classes.
  • Lens — integrated in nav.js. Will need its own pass.

What stays vanilla forever

  • shared.css base tokens — color vars, font vars. Tailwind config reads these.
  • theme.js — light/dark/accent attribute switcher. Tailwind reads attributes.
  • nav.js — header/sidebar injection. Migrates to Tailwind classes during own pass.
  • JS render functions — they emit HTML strings with class names. Class definitions move to Tailwind/component layer; class names stay so JS doesn't change.
